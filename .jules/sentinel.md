## 2026-03-18 - [DoS via Memory Exhaustion via Large Payloads on Unhandled Methods]
**Vulnerability:** The `PayloadSizeLimitMiddleware` previously restricted incoming request payload sizes only for explicitly defined methods (`POST`, `PUT`, `PATCH`). This allowed attackers to send arbitrary large bodies via unexpected methods (e.g., `GET`, `DELETE`, `OPTIONS`), bypassing the size limit and causing potential memory exhaustion (DoS).
**Learning:** Security controls like payload size limiters should default to securely processing all methods rather than whitelisting known data-carrying methods, because an attacker can exploit the unhandled branches.
**Prevention:** Apply ASGI payload size limits uniformly across all `http` request methods. Ensure no assumptions are made about which methods can or cannot carry a body, as HTTP/1.1 allows bodies on almost any method, and ASGI servers will process them.
## 2026-03-19 - [DOM-based XSS via innerHTML]
**Vulnerability:** The frontend application used `innerHTML` to dynamically inject HTML strings constructed from user input (`stackInput.value`). Even though the input was split and checked for numbers, the initial DOM element creation logic concatenated un-sanitized values (`angle`) into HTML tags (``<span ...>${angle}°</span>``), leading to a DOM-based XSS vulnerability.
**Learning:** Never trust client-side validation alone to sanitize inputs that are directly injected into the DOM via `innerHTML`. Even if inputs appear to be "validated" as numbers or limited strings, raw HTML concatenation remains a prime vector for XSS if an attacker can bypass those checks or inject unexpected payloads.
**Prevention:** Construct DOM elements securely using `document.createElement()` and `textContent` or `document.createTextNode()`. This ensures that user input is treated as text data rather than executable HTML, completely mitigating the risk of DOM-based XSS.
